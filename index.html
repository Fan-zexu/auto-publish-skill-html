📋 项目概述

🎯 项目定位

这是一个面向美团外卖广告B端前端团队的 AI Agent Skill，以「AI 驱动的 Cron 定时任务 + 多 Skill 编排」的方式，将传统手工发版流程完全自动化。每周一至周四按固定节奏自动执行 Stage/Prod 全链路发版流程，值班人仅需在关键节点做人工确认。

💡 解决的核心痛点

传统发版流程中，值班人需要手动逐一检查 20+ 前端应用的 PR 合入状态、逐个触发 Stage 部署、挨个催促未完成备机验证的同事、手工汇总上线计划信息。整个流程每次耗时 2-3 小时且高度重复，容易遗漏应用或 @错人。本系统将这些环节全部自动化，值班人从「执行者」变为「审批者」。

🧩 Skill 编排模式

系统采用「主 Skill + 多关联 Skill」的微服务编排模式。主 Skill（adfe-b-publish-automation）作为编排控制器，负责流程调度、状态管理和异常处理；关联 Skill 各自封装单一职责的能力（查询上线计划、发送大象消息、操作部署平台等）。这种设计实现了关注点分离，每个 Skill 可以独立迭代和复用。

🔄 核心发版流程（4 Step 流水线）

整个发版周期以「周一/三 → 周二/四」为节奏，分 4 个阶段自动执行。每个阶段通过 Cron 定时触发，触发后先校验值班人身份，匹配则执行，不匹配则静默跳过。

周一/三 11:00 · 12:00 · 14:00 · 15:00
Step 1：检查 PR 合入状态 + 催促 RD
调用 adfe-b-launch-plan-info 获取上线计划详情 → 提取每个前端应用的「合入集成分支」状态 → 对 jobName 做前缀白名单二次校验 → 筛选未合入应用 → 对其前端 RD 做 fe-rd-list.txt 交集过滤 → 批量查询 UID 和中文姓名 → 通过 daxiang-sender 在发版群中 @ 未合入的 RD 进行催促。若全部合入则私聊通知值班人，同一天只通知一次（状态去重）。
周一/三 16:00
Step 2：自动部署 Stage 环境
获取上线计划 + Groot 项目列表 → 将前端应用分为三类：A（Groot 部署）、B（Plus 平台）、C（未录入）→ 私聊通知值班人部署概况 → 对 A 类项目先执行 dry-run 预检 → 通过预检的逐个正式部署（间隔 3s 防并发）→ 轮询部署状态（30s/次，最长 30min）→ 汇总结果私聊通知值班人。
周二/四 11:00 · 12:00 · 14:00
Step 3：检查备机验证 + 催促 RD
与 Step 1 同构但检查维度不同：提取每个应用的「备机验证」状态 → 非 ✅ 的视为未完成 → 催促未完成应用的前端 RD → 全部完成则私聊通知值班人。同样有状态去重和前端 RD 过滤机制。
周二/四 15:30
Step 4：发版前确认 — 私聊上线计划详情
获取完整上线计划详情 → 在消息头部添加发版时间窗口提醒（解耦项目 14:00-17:00、其他项目 20:00 后）→ 通过 Markdown 格式私聊发送给值班人，值班人人工核查后手动触发 Prod 发版。

📨 消息发送的「文件中转」模式

这是一个从历史事故中提炼出的核心设计模式。大象 IM 的 @ 协议文本格式为 [@姓名|mtdaxiang://profile?uid=xxx]，包含 |、://、[、] 等 Bash 特殊字符。如果直接 inline 到 shell 命令行的 --text 参数中，Bash 会错误解析导致消息被截断。

解决方案是「三步文件中转法」：先用 write 工具将消息写入临时文件 → 再写入 at 列表文件 → 最后通过一个 Python 中间脚本读取文件内容，以 subprocess.call 传给 send.py，消息内容完全不经过 Bash 解析。

# 1. 消息内容写入文件（不经过 bash） write → /tmp/msg_step1.txt # 2. @ 列表写入文件 write → /tmp/at_list.txt # 每行 uid:姓名 # 3. Python 中间脚本读文件 → 调用 send.py import subprocess with open(msg_file) as f: text = f.read().strip() cmd = ['python3', send_py, 'send', '--group', group, '--client-id', dx_cid, '--text', text, '--custom-at'] subprocess.call(cmd) # 不用 capture_output
🔍 前端 RD 双重过滤机制

edopenapi 接口返回的 rd 数组包含前端、后端、测试、PM 所有角色。如果不过滤就直接 @ 到群里，后端同事会被频繁打扰——这是历史上最常犯的错误。

系统实施了两道过滤防线：第一道由 adfe-b-launch-plan-info Skill 在数据源层完成（edopenapi rd ∩ fe-rd-list.txt）；第二道由本 Skill 在消费层做 jobName 前缀白名单二次校验。即使上游 Skill 出现 bug 返回了非前端应用，也会被拦截。

💾 状态管理与幂等设计

系统通过 state/daily-state.json 实现通知幂等。例如"全部合入"通知，在首次发送后记录当天日期到 allMergedNotifiedDate 字段，后续轮次检测到日期匹配则跳过，避免同一天重复通知值班人。文件持久化在容器外部，不受重启影响。文件读取失败时默认视为"未通知"，保证了容错性。

🚀 Stage 部署的 dry-run + 轮询模式

部署 Stage 环境时采用「预检 → 正式部署 → 轮询」三段式策略。先对所有 A 类项目执行 groot publish --dry-run 预检，预检失败的项目直接标记跳过。正式部署时逐个触发并间隔 3 秒防并发冲突，从 stdout 提取 logInfo.id 用于后续状态查询。部署完成后以 30 秒间隔轮询所有项目的部署状态，最长 30 分钟超时，全部到达终态后一次性汇总通知值班人。

🚨 五条绝对红线

系统对 AI Agent 的行为做了强约束，这在 AI Agent 工程化中是非常关键的设计：

#红线原因
1消息格式逐字照抄模板自创文案曾导致发版群收到不规范的通知
2RD 数据只来自接口查询AI 曾编造 RD 信息导致错误 @
3@ 显示名必须用中文姓名用 misId 显示不友好且对方难以识别
4前端 RD 过滤不可跳过后端 RD 被 @ 造成严重干扰（最常犯的错）
5消息内容不可 inline 到 shell@ 协议特殊字符被 Bash 截断（历史事故）
🔄 自愈机制

系统具备多种自动恢复能力：fsd-sso Token 过期时自动执行 fsd-sso login 重新认证；Groot Token 失败时自动重新获取 Client Credentials；mtcli-kms 未安装时自动从 npm 安装；命令版本不满足要求时自动升级。这些都属于「可预判的环境问题」，无需人工介入。只有在自动修复尝试失败后，才通过私聊通知值班人。

🧪 Preflight Checklist 机制

每次执行「发群消息」前，AI 必须逐条读取 references/preflight-check.md 中的自检项，全部确认通过后才能调用 send.py。这份 checklist 是从历史事故中提炼出来的，包含了消息格式、RD 数据源、@ 协议完整性等所有容易出错的点。这是一种「将事后复盘制度化」的工程实践。

🎤 面试话术参考

Q：请介绍一下这个项目的背景和你解决了什么问题？
我们外卖广告前端团队有 20+ 个前端应用，每周发版两次。传统发版流程中，值班人需要手动检查每个应用的 PR 合入状态、逐个催促未合入的同事、手动触发 Stage 部署、检查备机验证、汇总上线计划——整个流程每次耗时 2-3 小时，且全靠人工操作容易遗漏。我设计并实现了一个 AI Agent Skill 驱动的全流程自动化发版系统，通过 9 个定时任务覆盖周一到周四的完整发版周期，将值班人从「执行者」解放为「审批者」，发版工作量接近零干预。
Q：你在这个项目中遇到的最大技术挑战是什么？
最大的挑战是 AI Agent 的行为可靠性。AI 在发群消息时天然有「自作聪明」的倾向——改文案措辞、跳过它认为不必要的步骤、编造数据。我们早期就出过 AI 把后端 RD 也 @ 进来的事故。我设计了一套「五条绝对红线 + preflight checklist + 双重 RD 过滤」的防错体系：消息模板逐字锁定、前端 RD 列表做两道交集过滤、消息发送前强制执行自检清单。这套机制本质上是在解决「如何让 AI Agent 在生产环境中可控可靠地工作」这个业界普遍面临的难题。
Q：系统的认证设计是怎么考虑的？
系统涉及四套认证体系，最复杂的点在于沙箱环境的凭证优先级覆盖问题。AI Agent 运行在沙箱环境中，沙箱会预设 DX_CLIENT_ID 环境变量指向沙箱 Bot。但我们需要用 Groot 机器人的凭证发群消息。如果用 --config 方式传入凭证，会被环境变量覆盖导致"机器人不在该群"错误。我的解决方案是从 KMS 动态读取凭证，通过 --client-id 命令行参数显式传入（最高优先级），同时在每次执行前做凭证预检——用 Groot 凭证调一次 query-uid.py 确认可用。
Q：你是怎么设计这个系统的架构的？
我采用了「主 Skill 编排 + 多关联 Skill 微服务」的架构。主 Skill 是编排控制器，负责流程调度、状态管理和异常处理；6 个关联 Skill 各自封装单一职责（上线计划查询、大象消息发送、部署平台操作等）。这种设计的好处是：一、关注点分离，每个 Skill 可以独立迭代；二、能力复用，比如 daxiang-sender 也被其他团队使用；三、故障隔离，某个 Skill 出问题不影响其他。调度层用了 9 个 Cron 定时任务，每个任务使用 --session isolated 确保隔离执行，互不干扰。
Q：这个系统上线后效果如何？
上线后的效果主要体现在三个维度。效率提升：值班人发版工作量从每次 2-3 小时降至接近零干预，只需在 Step 4 私聊收到的上线计划详情中做人工确认。质量提升：通过双重 RD 过滤和 preflight checklist，杜绝了之前频繁出现的"@ 错人"问题。覆盖度：系统经历了 V56 次迭代打磨，覆盖了各种边界情况（某个应用无前端 RD、凭证过期、网络超时、上线计划为空等），已经成为团队每周发版的核心基础设施。
