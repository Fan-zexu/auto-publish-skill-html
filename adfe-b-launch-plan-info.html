📋 项目概述

🎯 定位与职责

adfe-b-launch-plan-info 是自动化发版系统的核心上游数据源。主 Skill（adfe-b-publish-automation）在 Step 1（检查合入）和 Step 3（检查备机验证）中都通过调用此 Skill 获取所有前端应用的详细状态——包括应用名称、前端 RD、合入集成分支状态、备机验证结果、最新 Tag。

它的核心价值在于将三个分散的数据源（edopenapi 应用列表、fsd pub status 合入状态、fsd pub pre-rtag 备机验证）聚合为一张统一的结构化表格，同时实施双重过滤（前缀白名单 + 前端 RD 交集），确保只有前端应用和前端 RD 进入下游流程。

💡 解决的问题

如果没有这个 Skill，主 Skill 需要自己分别调用三个数据源、手动拼接数据、手动过滤非前端应用和非前端 RD——这正是历史上多次出错的根源。将数据聚合和过滤逻辑封装到独立 Skill 中，实现了「数据层一次正确过滤，消费层零负担使用」。

🧩 与主 Skill 的数据流

主 Skill 禁止绕过此 Skill 直接调用 edopenapi 或 fsd CLI。原因是此 Skill 已完成两道过滤（jobName 前缀 + fe-rd-list.txt 交集），绕过后需要手动重现这两道过滤，历史上多次因此出错。如果此 Skill 报错或超时，应修复后重试，而不是换路走。

🔄 数据聚合流程

Step 1
获取上线计划 ID
接收传入的 PLAN_ID 参数，或调用 adfe-b-launch-plan Skill 搜索当天的上线计划。获取到 PLAN_ID 后进入数据查询阶段。
Step 2
edopenapi 获取应用列表
使用 mtsso-skills-official 注入的 ${user_access_token} 构造 Cookie，调用 getListByOnliProgramId 接口获取上线计划下所有应用的名称、jobName、RD 数组、最新 Tag 等信息。
Step 3
并行查询合入集成分支状态
对每个前端应用并行执行 fsd pub status -i <PLAN_ID>，提取「合入集成分支」列的状态（✅ success / ❌ fail / ⏳ pending / N/A）。失败时最多重试 5 次，间隔 2 秒。
Step 4
并行查询备机验证结果
对每个前端应用并行执行 fsd pub pre-rtag -i <PLAN_ID>，提取备机验证状态。同样最多重试 5 次，间隔 2 秒。
Step 5
数据汇总与过滤
按 jobName 关联三个数据源的结果 → 执行前缀白名单过滤（5 个前缀）→ 执行 fe-rd-list.txt 全等交集过滤 → 检测接口返回的结构变化（缺失字段降级为 N/A 并追加警告）→ 输出概览 + Markdown 表格。

⚙️ 关键技术细节

🔍 双重过滤机制

第一道：jobName 前缀白名单 — 只有 jobName 以下列前缀开头的应用才被视为前端应用：

FE_PREFIXES = [ 'waimai_ad_fe', 'waimai_cd_fe', 'mach_pro_waimai_ad', 'sailor_fe', 'adfe' ]

第二道：fe-rd-list.txt 全等交集 — edopenapi 返回的 rd 数组包含前端+后端+测试+PM 所有角色。必须与 fe-rd-list.txt 名单取全等交集（=== 比较），只保留前端 RD。这是防止后端同事被 @ 的核心防线。

🛡️ 结构变化检测

edopenapi 接口的返回字段可能随时变更（字段重命名、新增、删除）。此 Skill 在解析响应时对每个预期字段做存在性检查，字段缺失时不报错崩溃，而是降级为 N/A 并在输出末尾追加结构变化警告。这保证了即使接口更新，Skill 也能输出有效（尽管可能不完整）的结果，而不是直接中断发版流程。

🔄 通用重试机制

fsd CLI 命令受网络抖动影响较大。Step 3 和 Step 4 对每个 fsd pub status / fsd pub pre-rtag 调用实施最多 5 次重试、间隔 2 秒的策略。重试耗尽仍失败的应用，其对应列标记为 N/A 而非中断整个查询。

🎤 面试话术参考

Q：这个模块是怎么聚合多数据源的？
这个 Skill 需要聚合三个异构数据源：edopenapi 的应用列表和 RD 信息、fsd pub status 的合入集成分支状态、fsd pub pre-rtag 的备机验证结果。我的设计是先用 edopenapi 获取应用基础信息作为主数据，然后并行调用两个 fsd CLI 命令获取状态数据，最后按 jobName 做关联拼接。关键的工程挑战有两个：一是 fsd CLI 的网络稳定性不够，我实现了通用重试机制（5 次重试、2s 间隔），失败的应用降级为 N/A 而不是中断整个查询；二是 edopenapi 的返回结构可能变更，我加入了结构变化检测，字段缺失时降级处理并追加警告，保证上游接口变化不会导致发版流程崩溃。
Q：前端 RD 过滤为什么需要两道防线？
这是从真实事故中总结出来的。edopenapi 返回的 rd 数组包含了所有角色——前端、后端、测试、PM。如果不过滤就直接 @ 到群里，后端同事会被频繁打扰，这是历史上最常犯的错误。第一道防线是 jobName 前缀白名单（5 个前缀），确保只有前端应用进入流程；第二道防线是 fe-rd-list.txt 全等交集，用一份维护好的前端 RD 名单对每个应用的 rd 数组做过滤。两道防线的设计思想是深度防御：即使名单文件更新滞后漏掉了新入职的前端，前缀过滤也能保证不会把后端应用的 RD 拉进来；反过来，即使前缀规则不够精确，名单交集也能确保最终 @ 的都是前端同事。
Q：为什么主 Skill 禁止绕过此 Skill 直接查数据？
这是一个架构治理决策。此 Skill 封装了两道过滤逻辑（前缀 + RD 交集），如果主 Skill 绕过它直接调用 edopenapi 和 fsd CLI，就需要在主 Skill 中重新实现这两道过滤。历史上确实发生过主 Skill 绕路后忘记过滤导致 @ 错人的事故。所以我在文档中明确写了「禁止绕过」的约束——单一数据出口原则，所有消费方只能通过此 Skill 获取已过滤的数据，从源头保证数据正确性。
