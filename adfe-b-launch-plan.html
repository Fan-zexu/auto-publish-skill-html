📋 项目概述

🎯 Skill 定位

adfe-b-launch-plan 是整个自动化发版流程的 前置依赖。在每次发版流程启动时，系统首先需要定位当天的上线计划 ID，才能进行后续的 PR 合入检查、Stage 部署、备机验证等环节。本 Skill 通过标准化的搜索关键词，在 FSD 发版平台中精准匹配到目标上线计划，并返回计划 ID、URL 及状态等基本信息。

💡 核心理念

「所有判断逻辑由 Shell 脚本执行，AI 仅根据 exit code 决定分支」—— 这是本 Skill 的第一设计原则。日期推算、关键词构建、搜索结果匹配，这些看似简单却极易出错的环节全部交给确定性的 Shell 脚本处理。AI 的职责被严格限定为：调用脚本、读取 exit code、根据预定义的分支走向执行下一步。这种设计彻底杜绝了 AI 自行推算日期导致的偏差（例如把「周四」推算成错误的日期）。

🧩 分层设计思路

架构按「推算 → 构建 → 搜索 → 提取」四层解耦。每一层由独立的 Shell 脚本负责，通过标准输出和 exit code 进行层间通信。这种设计的好处是：每层可以独立测试和调试，且 AI 在任何一层都不参与实质性的计算或判断 —— 它只是串联各层的「调度员」。

🔄 核心流程

搜索流程采用「CLI 优先 + 浏览器降级」的双路径策略，所有判断逻辑通过 Shell 脚本的 exit code 驱动分支。

Step 1 · 日期推算
calc_date.sh 推算目标上线日期
接收用户输入的时间描述（如"周四""明天""下周一"等中文关键词），由 Shell 脚本进行确定性的日期计算，输出标准格式的目标上线日期。AI 不参与任何日期推算逻辑，仅传递用户输入并读取脚本输出。
Step 2 · 关键词构建
build_keyword.sh 生成搜索关键词
将 Step 1 输出的日期拼接为标准化搜索关键词：【商业增值】【商家端】-{DATE}-上线计划。关键词格式固定，由脚本硬编码，不依赖 AI 生成或拼接。
Step 3 · CLI 精确匹配
fsd pub list + match_plan.sh 精确匹配
通过 fsd pub list 命令在 FSD 平台搜索上线计划列表，将结果传入 match_plan.sh 进行精确匹配。脚本通过 exit code 返回匹配结果：exit 0 = 匹配成功，exit 1 = 未匹配。AI 仅读取 exit code 决定后续分支走向。
Step 4a · 匹配成功
fsd pub info 获取详情
当 match_plan.sh 返回 exit 0 时，从匹配结果中提取计划 ID，调用 fsd pub info 获取上线计划的详细信息，包括计划 ID、URL、状态、关联应用列表等。
Step 4b · 匹配失败降级
浏览器自动化访问 FSD 搜索页
当 match_plan.sh 返回 exit 1（CLI 未匹配到目标计划）时，降级为浏览器自动化路径：通过 agent-browser 打开 FSD 发版平台的搜索页面，使用构建好的关键词进行页面级搜索。此路径耗时更长（秒级），但可以处理 CLI 命令未覆盖的边界情况。

⚙️ 关键技术细节

🔧 Shell 驱动的确定性设计

本 Skill 的核心设计哲学是「把 AI 从判断者降级为调度者」。所有涉及计算、匹配、判断的逻辑全部封装在 Shell 脚本中：

环节Shell 脚本职责AI 职责
日期推算calc_date.sh 解析中文关键词，输出标准日期传递用户输入，读取 stdout
关键词构建build_keyword.sh 按固定模板拼接传递日期参数，读取 stdout
结果匹配match_plan.sh 精确匹配，返回 exit code读取 exit code，走预定义分支

这种设计的价值在于：Shell 脚本的行为是 100% 确定性的，不会因为 prompt 的微小变化而产生不同结果。而 AI 的日期推算天然存在不确定性 —— 例如"下周四"在不同上下文中可能被理解为不同日期。

🔀 双路径搜索策略

系统设计了 CLI 快速路径 和 浏览器降级路径 两条搜索通道：

维度CLI 快速路径浏览器降级路径
工具链fsd pub list + match_plan.shagent-browser
响应速度毫秒级秒级

这种设计遵循「快路径优先、慢路径兜底」的工程原则。绝大多数情况下 CLI 路径即可完成匹配，浏览器路径仅在 CLI 返回 exit 1 时作为最后手段。

# 关键词模板（由 build_keyword.sh 硬编码） 【商业增值】【商家端】-{DATE}-上线计划 # 示例输出 【商业增值】【商家端】-2026-01-15-上线计划 # 完整调用链 user_input="周四" date=$(./calc_date.sh "$user_input") # → 2026-01-15 keyword=$(./build_keyword.sh "$date") # → 【商业增值】【商家端】-2026-01-15-上线计划 fsd pub list --keyword "$keyword" | ./match_plan.sh # exit 0 or 1

🎤 面试话术参考

Q：为什么不让 AI 直接推算日期？
这是一个确定性 vs 概率性的设计选择。AI 的日期推算本质上是概率性的 —— 例如用户说"周四"，AI 需要判断是「本周四」还是「下周四」，而这取决于当前是周几、发版节奏是什么、是否考虑节假日。早期版本确实让 AI 直接推算，结果出现过把"周四"理解为错误日期的情况，导致搜索到了错误的上线计划。解决方案是将所有日期推算逻辑封装到 Shell 脚本（calc_date.sh）中，脚本内部用确定性的 date 命令计算，AI 只负责传参和读结果。这个设计原则后来被推广到整个搜索链路 —— 关键词构建、结果匹配全部由 Shell 脚本完成，AI 降级为纯粹的「调度员」。这种做法实际上是在回答一个更大的问题：AI Agent 系统中，哪些环节适合用 AI，哪些环节必须用确定性代码。
Q：双路径搜索策略的设计思路是什么？
设计灵感来自网络请求的 fallback 机制。CLI 路径（fsd pub list + match_plan.sh）速度快、资源消耗低，适合处理 90% 以上的常规情况。但 CLI 存在局限性 —— 比如 FSD 平台 API 的搜索语义和页面搜索可能不完全一致，某些边界情况下 CLI 匹配不到但页面能搜到。所以我设计了浏览器降级路径作为兜底：当 match_plan.sh 返回 exit 1 时，系统自动切换到 agent-browser 打开 FSD 搜索页。核心思路是「快路径保效率、慢路径保成功率」。具体实现上，分支判断完全基于 exit code，不依赖 AI 分析搜索结果是否「看起来匹配」—— 这又回到了确定性设计原则：匹配与否由脚本判断，AI 只看 0 还是 1。
